## Overview

Object discussions on iOS currently have no notification preferences UI and no unread-state affordance. Users cannot control whether a given discussion pushes them every new reply vs only mentions, and they cannot tell at a glance that a document they're looking at has unread comments.

This plan delivers a **reduced first iteration** of IOS-6028: the in-object surfaces (notification preferences menu, Comments button blue dot, open-at-first-unread). The broader aggregate surfaces (Unread widget, Vault preview, pinned widget badge) are deferred because middleware does not yet propagate unread state to the parent object — only onto the derived chat object's details.

Middleware work merged 2026-04-21:
- GO-7168 PR #3090 — `notificationSubscribers` string-list relation on chat object + three RPCs (`ChatAddNotificationSubscriber`, `ChatRemoveNotificationSubscriber`, `ChatGetNotificationSubscribers`). Creator auto-added on object creation. Auto-subscribe on @mention is NOT implemented middleware-side; we skip it client-side.
- GO-7168 PR #3107 — `unreadMessageCount` / `unreadMentionCount` bundled relations on the chat object (typeKey `ot-discussion`).
- GO-7166 PR #3104 — `messageCount` response field + `ChatUpdateMessageCount` event. Total count, not unread. Not needed for this iteration.

## Solution Overview

**Chunk 1** adds the two-option Notifications submenu to the discussion 3-dot menu. Current mode is derived from whether the current user's identity is present in the chat object's `notificationSubscribers` list. Toggling calls the new middleware RPCs. Menu is hidden for Viewers.

**Chunk 2** adds the unread-awareness UX inside an object: a blue dot on the Comments/Discuss button when the chat has unread messages, and scroll-to-first-unread behavior when opening a discussion that has unread comments. Push notification tap routing is verified/patched so tapping a discussion push deep-links into the correct object's discussion screen scrolled to the referenced comment.

## Technical Details

### Notification preference state (Chunk 1)
- Source of truth: `notificationSubscribers: [String]` on the chat object (middleware-maintained).
- Current mode derivation:
  - `currentIdentity ∈ notificationSubscribers` → `.allNewReplies` (checked)
  - `currentIdentity ∉ notificationSubscribers` → `.mentionsOnly` (checked)
- Toggle actions:
  - Select "All New Replies" while in "Mentions Only" → `ChatAddNotificationSubscriber(chatObjectId, currentIdentity)`
  - Select "Mentions Only" while in "All New Replies" → `ChatRemoveNotificationSubscriber(chatObjectId, currentIdentity)`
  - Select currently-checked option → no-op.
- Current user's identity is already available from the session / participant service; find existing usage pattern during implementation.

### Unread subscription (Chunk 2)
- Given parent object's `chatId`, subscribe to that single chat object's details via the existing `objectIdsSubscriptionService` pattern.
- Request fields: `unreadMessageCount`, `unreadMentionCount`, optionally `notificationSubscribers`.
- Publish `hasUnread: Bool` ( = `unreadMessageCount > 0`) to the bottom navigation panel VM.

### Open-at-first-unread (Chunk 2)
- `DiscussionMessagesStorage.loadInitialMessages()` currently loads the last page. Modify initial-load path so:
  - If `ChatState.UnreadState.oldestOrderID` is non-empty at first-message arrival, call the existing `loadAroundOrderId(oldestOrderID)` path and publish an "unread divider" anchor before that message.
  - Else keep current "scroll to bottom" behavior.
- The `ChatState` arrives via the same `Chat.SubscribeLastMessages` response that already drives `DiscussionMessagesStorage` — no new subscription.

### Push notification tap routing (Chunk 2)
- Probe first: trigger a discussion push via TestFlight/simulator (or inspect existing handler). Most of the routing should already work thanks to IOS-5838's unified format + space→object→scroll-to-comment deep-link pipeline. Patch only if broken.
